## 🧪 Case Study: Настройка баланса RAG (Temperature & Prompt Engineering)

Одной из главных проблем RAG-систем является поиск баланса между **достоверностью** (отсутствием галлюцинаций) и **полезностью** ответа. В рамках проекта было проведено исследование влияния параметра `temperature` и структуры системного промпта на поведение модели GigaChat.

### Проблема: «Слишком честный, но бесполезный ответ»
**Настройки:** `temperature = 0.1` + Жесткий системный промпт (*«Отвечай строго по тексту, не придумывай от себя»*).
При таких настройках модель идеально защищена от галлюцинаций, но теряет способность делать логические выводы, если ответ в тексте не сформулирован прямо.

* **Вопрос пользователя:** *«Для чего нужен синтаксический сахар в декораторах?»* (В базе был только пример кода с декоратором и упоминание самого термина).
* **Ответ агента:**
  > 🤖 В представленном контексте нет информации о синтаксическом сахаре в декораторах Python... Следовательно, исходя исключительно из данного контекста, невозможно ответить на вопрос о его предназначении.

**Итог:** Факт галлюцинации равен 0%, но проблема пользователя не решена.

### Решение: «Безопасная экспертность»
**Настройки:** `temperature = 0.2` + Гибкий системный промпт (*«Если информации нет, честно скажи об этом, но попробуй дать краткий ответ на основе своих общих знаний, явно уточнив источник»*).
Мы дали модели чуть больше креативности для связки слов и разрешили использовать внутренние веса при условии строгой маркировки.

* **Тот же запрос:** *«Для чего нужен синтаксический сахар в декораторах?»*
* **Ответ агента:**
  > 🤖 В моих документах контекста про синтаксический сахар именно в декораторах нет, зато общие сведения имеются.
  > 
  > **Синтаксический сахар** — это элементы синтаксиса языка, которые делают запись программы более удобной и выразительной. В Python декораторы позволяют добавлять функциональность к функциям без изменения их исходного кода.
  > *(Далее модель самостоятельно привела корректный пример кода с `@my_decorator`)*

**Вывод:** Минимальное изменение `temperature` с 0.1 до 0.2 в связке с грамотным Prompt Engineering превратило систему из «сухого поисковика» в полноценного интеллектуального ассистента, который сохраняет честность (не выдает свои знания за документы компании), но при этом максимально полезен пользователю.


## 🛠 Стек технологий (Tech Stack)
Стек подобран с учетом современных стандартов разработки AI-приложений (AI-Native) и требований к масштабируемости.

* **Core Backend:** Python 3.10+ (Строгая типизация, ООП).
* **LLM Orchestration:** LangChain (Использование современного синтаксиса **LCEL** — LangChain Expression Language для построения цепочек).
* **AI & NLP:** * **LLM:** GigaChat API (Интеграция с коммерческими моделями).
  * **Embeddings:** HuggingFace `intfloat/multilingual-e5-base` (Использование локальных open-source моделей для векторизации текста, что экономит API-запросы).
* **Vector Database:** Qdrant (Современная и быстрая векторная СУБД для семантического поиска).
* **Data Processing:** `PyPDF`, `python-docx` (ETL-процессы: извлечение неструктурированных данных из файлов).
* **Security & Auth:** `python-dotenv` (безопасное хранение секретов), кастомная логика ротации токенов.

## 🏗 Архитектура приложения (Data Flow)
Проект спроектирован с упором на модульность и разделение ответственности (Separation of Concerns). Приложение состоит из двух независимых пайплайнов: **Индексации (ETL)** и **Генерации (RAG)**.

### 1. Data Ingestion Pipeline (Подготовка базы знаний)
Модуль `DocumentIngestor` отвечает за наполнение векторной базы:
* **Extract:** Автоматическое сканирование директории и чтение PDF/DOCX/TXT файлов через специализированные лоадеры.
* **Transform:** Разбиение больших текстов на смысловые фрагменты (Chunks) с помощью `RecursiveCharacterTextSplitter` (настроен *chunk_size=1000* и *overlap=200* для сохранения контекста на стыке абзацев).
* **Load:** Преобразование текста в многомерные векторы (Embeddings) и их сохранение в коллекцию **Qdrant**.

### 2. Retrieval-Augmented Generation Pipeline (Поиск и Ответ)
Модуль `RAGChain` обрабатывает запросы пользователя в реальном времени:
* **Semantic Retrieval:** Модуль `DocumentRetriever` переводит текстовый вопрос пользователя в вектор и с помощью косинусного сходства (similarity search) находит в Qdrant Top-5 наиболее релевантных фрагментов текста.
* **Prompt Engineering:** Найденные фрагменты «склеиваются» и подставляются в системный промпт (Prompt Template) вместе с исходным вопросом. Промпт содержит строгие инструкции по предотвращению галлюцинаций.
* **LLM Generation:** GigaChat получает обогащенный контекстом запрос и генерирует точный, полезный ответ, опираясь исключительно на предоставленные факты.

### 🔐 Инфраструктурный модуль: Token Management
Модуль `GigaChatTokenManager` работает в фоне и решает архитектурную проблему истекающих авторизационных сессий (OAuth2). Он валидирует временные метки (timestamp) и превентивно запрашивает новый токен за 5 минут до истечения старого, обеспечивая 100% стабильность сервиса.
